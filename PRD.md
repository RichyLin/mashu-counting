# 麻薯计分 — 产品需求文档

**版本：** 1.2  
**日期：** 2026-04-25  
**状态：** 已上线（https://mashu-counting.vercel.app）

---

## 一、产品概述

### 1.1 产品定位

麻薯计分是一款面向**棋牌类游戏**的 H5 网页计分工具，不限于特定玩法。玩家在同一局游戏中通过手机浏览器实时记录分数，避免口头记分混乱。

### 1.2 目标用户

以朋友聚会场景为主，适用于任何需要多人实时计分的棋牌游戏。

### 1.3 核心价值

- 实时同步：所有玩家看到的分数状态保持一致
- 操作简单：针对手机端设计，单手操作为主
- 有据可查：完整的交易记录，支持撤回

### 1.4 平台

- H5 网页，手机浏览器访问
- 设计基准宽度：375px（iPhone SE）
- 不开发原生 App
- 国内访问需绑定自定义域名（`vercel.app` 域名被 GFW 屏蔽）

## 二、名词定义

| 名词 | 说明 |
|------|------|
| 房间 | 一局游戏的虚拟空间，由 4 位数字编号标识 |
| 房主 | 第一个进入房间的玩家，拥有踢人、重置分数、解散权限 |
| 公池 | 所有玩家共享的分数池，交公池/收公池操作均作用于此 |
| 座位 | 麻将桌上的固定位置：上/左/右/下（下方固定为自己） |
| 转账 | 一名玩家将分数付给另一名玩家的操作 |

## 四、页面详细需求

### 4.1 入口页

**功能：**
- 居中展示产品 Logo（麻薯贱宝正视图站立 + 产品名「麻薯计分」）
- 两个主操作按钮：「创 建 房 间」和「加 入 房 间」
- 点击任一按钮后展示 4 位数字输入框 + 确认按钮 + 返回链接

**交互流程：**

```
初始状态
  ├── 点击「创建房间」→ 展示输入框（模式=创建）
  │       ├── 输入4位号码 → 确认
  │       │     ├── 号码已存在 → 内联错误提示「该房间号已存在，请加入或换一个号码」
  │       │     └── 号码不存在 → 创建成功 → 进入身份设置页
  │       └── 点击「返回」→ 回到初始状态（两个按钮）
  │
  └── 点击「加入房间」→ 展示输入框（模式=加入）
          ├── 输入4位号码 → 确认
          │     ├── 号码不存在 → 内联错误提示「该房间号不存在，请检查号码或创建新房间」
          │     └── 号码存在 → 进入身份设置页（若本设备已有身份则直接进主界面）
          └── 点击「返回」→ 回到初始状态（两个按钮）
```

**规则：**
- 输入框限制：只允许数字，限4位，不足4位不可提交
- 错误提示：内联显示在输入框下方，不使用 Toast
- 入口页**不展示副标题**，不绑定特定游戏类型

---

### 4.2 身份设置页

**功能：**
- 返回按钮（回到入口页）
- 头像选择区：展示 emoji 网格（5列×3行，共15个），点击选中，高亮已选项
- 姓名输入框
- 「进入房间」按钮（姓名为空或不合规时禁用）

**规则：**
- 头像默认选中第一个 emoji（🍡）
- 不支持上传照片或拍照，仅支持从预设 emoji 中选择
- 姓名限制：
  - 仅允许汉字、字母、数字，不允许特殊字符（空格、标点等）
  - 最多6个字符
  - 实时校验，不合规时显示错误提示并禁用提交按钮
- 进入后，第一个进入该房间的玩家自动成为房主
- 座位分配顺序：第1位→下，第2位→上，第3位→左，第4位→右

**身份持久化：**
- 用 localStorage 绑定设备 ID + 房间号，记录玩家 ID
- 同一设备再次访问同一房间，直接跳过身份设置页，恢复上次状态进入主界面

### 4.3 主界面（麻将桌）

**整体布局（从上至下）：**

```
┌─────────────────────┐
│  房间 XXXX   X人在线 │  ← 顶部导航栏（56px）
├─────────────────────┤
│         [上]         │
│  [左]  公池  [右]   │  ← 麻将桌区域（flex:1）
│         [下=我]      │
├─────────────────────┤
│    记录  |  设置     │  ← 底部导航栏（68px，设置仅房主可见）
└─────────────────────┘
```

**麻将桌区域：**
- 背景：淡绿色光晕
- 中央：抹茶绿椭圆形桌面（深棕描边）
- 公池区域位于桌面中央（见 4.3.1）
- 4个固定座位：上/左/右/下（见 4.3.2）

#### 4.3.1 公池区域

- 显示：麻薯贱宝侧躺图 + 金色分数框（「公池 X 分」）
- 初始显示「公池 0 分」
- 点击任意区域 → 打开公池操作弹窗
- 公池分数实时同步至所有玩家

#### 4.3.2 玩家座位

每个座位显示：
- 圆形 emoji 头像（其他玩家：深棕描边；自己：金色描边）
- 皇冠图标👑（仅房主自己的头像上方显示）
- 姓名标签（白底深棕描边胶囊形）
- 分数（**仅自己的分数对自己可见**，其他玩家分数不显示）

自己的座位元素顺序（从上至下）：
1. 圆形头像（金色描边，顶部显示皇冠👑）
2. 姓名标签（「姓名（我）」格式）
3. 分数（Fredoka 字体，大号）

空位规则：
- 不满4人时，空位显示**虚线圆圈占位符**（可作为换座目标）
- 4个座位位置固定，下方始终为本机玩家

视图旋转规则：
- 每个玩家看到的麻将桌中，自己始终在下方
- 其他玩家的座位按顺时针映射：底→右→上→左

#### 4.3.3 换座交互

**触发方式（手机端）：**
- 长按（300ms）其他玩家头像后拖动
- 移动超过8px但未满300ms：取消，不触发换座

**拖拽行为：**
- 长按触发后：源头像变透明（opacity 0.35），出现跟随手指的 ghost 头像
- 拖动至目标位置（有人或空位虚线圆均可）高亮显示
- 松手后弹出换座确认弹窗（显示双方头像和姓名；若目标为空位则只显示被移动者）
- 确认后：更新 DB 中玩家的 seat 字段，实时同步至所有玩家

**限制：**
- **自己的头像不可拖动**，也不可作为换座目标
- 支持将玩家拖至**空位**
- 支持两个有人座位之间互换

**防误触：**
- 头像元素设置 `touch-action: none`、`-webkit-touch-callout: none`，防止长按触发系统选择菜单或 emoji picker

### 4.4 转账弹窗（底部 Sheet）

**触发：** 点击（短按，< 300ms）其他玩家头像

**内容：**
- 收款方信息行：emoji头像 + 姓名标签（「付给对方」）
- 提示文案：「你将付出多少分给 XX？」
- 数字输入框（大号，居中）
- 「确认转账」按钮（樱花粉色）

**规则：**
- 方向固定：**我付给对方**（不支持反向）
- 金额：正整数，无下限（可导致余额为负）
- 确认后：自己分数 -N，对方分数 +N
- 生成一条历史记录：「我 → 对方 N分」

---

### 4.5 公池操作弹窗（底部 Sheet）

**触发：** 点击主界面公池区域

**内容：**
- 当前公池分数展示（金色大号）
- 「交公池（-1分）」按钮（嫩芽绿）
- 「收公池（+X分）」按钮（黄金色）

**规则：**
- 任何玩家均可操作公池（不限房主）
- 交公池：固定扣除操作者 1 分，公池 +1
- 收公池：操作者获得公池全部分数，公池归零
- 公池为 0 时，「收公池」按钮置灰禁用
- 两个操作均可导致余额为负（无限制）
- 操作后生成历史记录

---

### 4.6 历史记录弹窗（底部 Sheet）

**触发：** 点击底部导航「记录」

**内容：**
- 标题「交易记录」
- 按时间倒序排列的记录列表

**每条记录格式：**
```
阿福 → 老虎  -5    [撤回]
```
- 正数（收入）：绿色
- 负数（支出）：红色
- 公池相关：金黄色
- 已撤回记录：灰色，无撤回按钮，显示「已撤回」

**撤回规则：**
- 点击「撤回」→ 弹出二次确认弹窗
- 确认后：
  1. 原记录标记为「已撤回」（不删除，保留在列表中）
  2. 新增一条【撤回】记录在列表顶部（分数变动反向）
- 【撤回】记录本身不可再次撤回

### 4.7 房间设置弹窗（底部 Sheet，仅房主）

**触发：** 点击底部导航「设置」

**内容与规则：**

| 功能 | 说明 |
|------|------|
| 初始分数 | 可编辑，失焦后自动保存，影响重置后各玩家的起始分数 |
| 踢出玩家 | 列出所有其他玩家，每人有「踢出」按钮，需二次确认；被踢者立即跳回入口页 |

| 解散房间 | 清除所有数据，所有玩家退出房间，需二次确认，返回入口页 |

## 五、房间生命周期

### 5.1 创建

- 任意玩家输入4位房间号进入，若房间不存在则自动创建
- 第一个进入的玩家自动成为房主，获得 bottom 座位
- 进入逻辑：查找已有房间 → 不存在则创建 → 创建冲突则重试查找（防并发和网络抖动）

### 5.2 加入

- 其他玩家输入同一房间号进入，按加入顺序分配剩余座位（上→左→右）
- 最多4人

### 5.3 被踢出

- 被踢玩家通过实时订阅感知，立即跳转至入口页
- 本地存储的身份记录同步清除，需重新输入身份才能再次加入

### 5.4 自动解散

满足以下任一条件时，房间自动解散（通过 pg_cron 每5分钟检查）：

| 条件 | 判断依据 |
|------|---------|
| 所有玩家均已离线超过5分钟 | 所有玩家 `last_seen_at` > 90秒前 |
| 无人操作超过2小时 | `last_active_at` < 当前时间 - 2小时 |

解散后状态更新为 `dissolved`，所有在线玩家通过实时订阅感知并跳回入口页。

### 5.5 手动解散

房主在设置中点击「解散房间」，二次确认后立即解散。

## 六、分数规则

- 初始分数：默认200分（房主可在设置中修改）
- 分数无下限：可为负数
- 各玩家**自己的分数只对自己可见**，其他玩家分数不显示
- 公池分数对所有人可见

---

## 七、实时同步规则

以下操作通过 Supabase Realtime 实时同步至所有在线玩家：

| 操作 | 同步内容 |
|------|---------|
| 转账 | 双方分数变化、历史记录 |
| 交/收公池 | 操作者分数、公池分数、历史记录 |
| 撤回 | 相关分数恢复、历史记录更新 |
| 踢出玩家 | 被踢玩家跳回入口页，其他人座位更新 |
| 重置分数 | 所有人分数重置、历史记录清空 |
| 解散房间 | 所有人跳回入口页 |
| 换座 | 座位变更同步至所有玩家 |

**实时同步注意事项：**
- 踢人依赖 `REPLICA IDENTITY FULL`（需在 Supabase 中对 players 表执行）才能正确触发 DELETE 事件过滤
- 所有涉及分数的操作通过 Postgres RPC 原子执行，避免并发导致数据不一致

## 九、技术架构

| 层级 | 技术选型 |
|------|---------|
| 前端 | React 19 + Vite 8 |
| 路由 | React Router v7 |
| 实时通信 | Supabase Realtime（Postgres Changes） |
| 数据库 | Supabase PostgreSQL |
| 部署 | Vercel（SPA routing via vercel.json rewrites） |
| 身份标识 | localStorage 设备ID（无账号体系） |
| 定时任务 | Supabase pg_cron（自动解散房间） |

### 数据库表结构

```
rooms        — id, code(varchar4 unique), host_device_id, init_score,
               status(active/dissolved), last_active_at
players      — id, room_id, device_id, name, emoji, seat(bottom/top/left/right),
               score, is_online, last_seen_at, joined_at
transactions — id, room_id, from_player_id, to_player_id, amount, type,
               description, revoked, created_at
room_pool    — room_id, score（单行，随房间创建自动初始化）
```

### 原子操作（RPC）

所有涉及分数变更的操作通过 Postgres 函数原子执行：

| 函数 | 说明 |
|------|------|
| `perform_transfer` | 转账：双方分数 ±N + 记录 |
| `perform_pool_pay` | 交公池：玩家 -1，池 +1 + 记录 |
| `perform_pool_collect` | 收公池：玩家 +池分，池归零 + 记录 |
| `perform_revoke` | 撤回：反向恢复分数 + 新增撤回记录 |
| `perform_reset` | 重置：所有玩家分数归初始值，清空记录 |

### Supabase 特殊配置

```sql
-- 踢人实时同步（DELETE 事件携带完整行数据）
ALTER TABLE players REPLICA IDENTITY FULL;

-- 自动解散房间（每5分钟执行）
SELECT cron.schedule('cleanup-inactive-rooms', '*/5 * * * *', $$
  UPDATE rooms SET status = 'dissolved'
  WHERE status = 'active' AND (
    last_active_at < NOW() - INTERVAL '2 hours'
    OR (last_active_at < NOW() - INTERVAL '5 minutes'
        AND NOT EXISTS (
          SELECT 1 FROM players WHERE room_id = rooms.id
          AND last_seen_at > NOW() - INTERVAL '90 seconds'))
  );
$$);
```

## 十、非功能需求

- **响应速度：** 实时操作（转账、公池）同步延迟 < 500ms
- **并发：** 单房间最多4人同时操作，无需高并发设计
- **离线降级：** 网络断开时提示，操作禁用，重连后自动恢复
- **兼容性：** iOS Safari 15+、Android Chrome 90+
- **安全：** 无敏感数据，房间号作为简单访问控制（非加密）
- **域名：** 国内访问需绑定自定义域名，vercel.app 被 GFW 屏蔽

---

## 十一、不在范围内（Out of Scope）

- 用户账号/登录体系
- 多局历史回顾
- 游戏规则自动计分

- 头像照片上传（仅 emoji 选择）
- 好友系统、社交功能
